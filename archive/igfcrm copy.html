Case Study

Influencer Gifting CRM Relationship visibility for brand marketers

Brands used Influencer Gift Form to gift $1k+ packages across multiple campaigns, but relationship history lived in spreadsheets. I designed a Relationship Hub to surface total spend, gifting frequency, and tracked posts in one place.

B2B SaaS Shopify app
Role: Product Designer Output: Coded interactive prototype Status: In development

Outcome preview

  • • Shipped (prototype): Table + sidebar details + add-post flow
  • • Designed for: 50+ influencer relationships per brand
  • • Constraint: No reliable auto post tracking → manual URL capture
  • • Current use: Customer demos to validate MVP scope

Key insight

The product already contains the data — it just isn’t organized by influencer relationship.

Brands exported to spreadsheets to answer repeat questions (spend, frequency, “did they post?”). Consolidating by influencer entity turns scattered campaign orders into relationship visibility.

Problem

Relationship history across campaigns was invisible in-app, so teams tracked it manually in spreadsheets.

Goal

Enable fast comparison + relationship context without adding workflow complexity.

How might we

Help marketers decide who to gift next using relationship data already in the system.

What I designed

Relationship Hub (MVP)

  • •Influencer table for fast comparison (sort/filter)
  • •Sidebar details to review posts without losing table context
  • •Add post flow (URL → preview → save) to reduce manual entry
  • •Data model influencer entity + dedupe/merge logic

Why it works

It mirrors spreadsheet mental models (rows/columns) while adding relationship context brands couldn’t get from individual campaigns.

Key decisions

The tradeoffs that shaped the MVP.

1) Tables over cards

Scale test: 50+ rows Primary task: comparison

Decision: Table layout for scan + sort + side-by-side comparison at real scale.

  • • Cards looked clean at 5–10, but became slow to scan at 50+
  • • Tables align values, match spreadsheet workflows, and reduce clicks

2) Manual post capture (URL) over auto-tracking

API/resource constraint MVP speed

Option A: Automatic scraping

Ideal experience, but technically complex + ongoing maintenance.

Option B: Manual URL + preview

One user action, but feasible now and matches “copy link” workflows.

Decision: URL-based capture with auto metadata/thumbnail, so MVP ships without platform-wide scraping.

3) Sidebar over modal/new page

Preserve context Compare multiple influencers
Modal blocks table context → high open/close friction.
New page breaks comparison and adds navigation overhead.
Sidebar keeps table visible → fast compare + browse.

Decision: Sidebar detail view to review posts while keeping sort/filter and row context visible.

Reflection

What I’d repeat, what I’d change, and what’s next.

What I learned

  • •Scale testing early prevents polished dead-ends (cards failed at 50+).
  • •Comparison was the real job → table + sidebar supported it best.
  • •Constraints shaped MVP → URL capture delivered value without scraping.
  • •Data architecture mattered → influencer entity + dedupe made the UI possible.

Next bets

  • •Filters: last gifted date, content type, spend tiers
  • •Notes/tags: follow-ups, relationship stage, campaign grouping
  • •Reporting: export + rollup metrics for stakeholders

Current impact

The coded prototype is used in customer demos to validate workflow, MVP scope, and prioritization for development.
